## Workflow: Managing Private vs Public Transcripts

### When a Transcript is Generated

After running `update-transcripts`, new transcripts appear in both repositories at the root level.

### Making a Transcript Private

If a transcript contains sensitive information:

```zsh
# In private repo
cd ~/Dropbox/claude/claude-transcripts
mv project-name private/

# In public repo
cd ~/Dropbox/claude/claude-transcripts-public

cd ~/Dropbox/claude/claude-transcripts-public

```

If you have a curated transcript ready to share:

1. Ensure it's generated in `claude-transcripts-public` root (not in `private/`)
2. Review the content to ensure no sensitive information
3. Update the `index.html` if needed (or regenerate transcripts)
4. Commit and push

GitHub Pages will update automatically after a few minutes.

## Privacy Guidelines

### Always Keep Private
- Project names that reveal sensitive information
- File paths with personal/confidential details
- Error messages containing credentials or API keys
- Detailed implementation of proprietary systems
- Any work-related or client projects

### Safe to Make Public
- Learning experiments and tutorials
- Open-source contributions
- Generic utility development
- Demonstrations of Claude Code capabilities
- Projects created specifically for sharing

## Tips

1. **Default to Private**: When in doubt, keep transcripts private. You can always make them public later.

2. **Review Before Publishing**: Always review a transcript before making it public, even if you think it's safe.

3. **Use `private/` Directory**: The `private/` directory in both repos is git-ignored, so transcripts there will never be pushed to GitHub.

4. **Separate Repos**: Keep the two repositories separate - don't mix private and public transcripts in the same commit.

5. **Update Regularly**: Run `update-transcripts` regularly to keep archives current, but remember to review and organize new transcripts afterward.
